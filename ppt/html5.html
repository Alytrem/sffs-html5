HTML4 : bilan

HTML4 (avec CSS2 et javascript) permet déjà de faire beaucoup de choses. Les applications web n'ont pas attendues HTML5 pour naitre. Les webmails ou éditeurs de documents en ligne (office web apps, google docs...) deviennent très évolués, le chargement dynamique avec Ajax devient usuel, les API comme Jquery fleurissent et rendent le développement plus rapide et robuste.

Voyons ensemble ce que l'HTML5 peut apporter de plus !

Introduction

  • HTML5 : le successeur de HTML4
  • Fruit du travail du W3C (World Wide Web Consortium) et du WHATWG (Web Hypertext Application Technology Working Group).
  • Les objectifs :
    1. Unification du comportement entre les navigateurs
    2. Normalisation
    3. Développement de nouvelles fonctionnalités HTML, CSS et javascript
  • HTML5 est encore une ébauche, mais déjà supportée par beaucoup de navigateurs, y compris mobiles.

HTML5 : plus que de l'HTML

HTML5 ~= HTML5 + CSS3 + JS

La timeline HTML5

  • Mai 2011 : Last Call (dernier appel à commentaires avant de figer les spécifications (s'il n'y a pas de commentaires).
  • 2014 : fin de la définition du standard (tous les objectifs de compatibilité seront alors atteints)
  • Mais rien n'empêche son utilisation dès maintenant. Il faut seulement prendre les précautions nécessaires.

Les objectifs de la présentation

  • Présenter les nouveautés HTML5, et faire un retour d'expérience
  • Evaluer ces nouveautés sur différents critères :
    • Maturité : indique le degré d'évolution possible avant 2014. Plus c'est mature, moins c'est susceptible de changer.
    • Intérêt : mesure l'apport en termes de fonctionnalités (subjectif)
    • Sécurité : mesure des possibles problèmes de sécurité (données accessibles en local par un tiers, données en clair sur le réseau, données accessibles par un autre site, donne un accès au système).
    • Compatibilité (Mobile, IE, Firefox, Chrome, Safari) : la note est maximale si les 3 dernières versions majeures supportent la fonctionnalité, puis perd 25% par versions ne le supportant pas, et tombe à 0 si la prochaine version ne le supportera pas non plus.

  • Faire un point sur l'aspect technique de la mise en œuvre des fonctionnalités

Nouvelles balises

On peut classer les nouvelles balises en 3 groupes.

  • Les balises ayant pour but d'enrichir la sémantique.
  • Les balises ajoutant des fonctionnalités
  • Les nouveaux inputs (champs de formulaires)

Enrichissement de la sémantique

<div id="header">
<div class="article">
...

Voici le genre de choses que l'on peut voir fréquemment. Seulement, un div ne veut rien dire. Arrivent donc de nouvelles balises porteuses de sens.

L'autre nouveauté est que HTML5 ne fonctionne pas avec une hiérarchie de titre (de h1 à h6) comme HTML4, mais avec une imbrication de sections, chaque section faisant repartir les titres à 1 en son sein.

Note : vous pouvez cependant toujours continuer à utiliser du code comme ci-dessus.

Les balises

On trouve ainsi les balises :

  • header/footer : une en-tête/pied d'une section ou d'un document.
  • hgroup : sorte de wrapper pour un ou plusieurs titres (h?) en lien (comme un titre et un sous-titre).
  • section : définition d'une nouvelle section.
  • article : similaire à section, mais plus riche sémantiquement.
  • nav : indique un menu
  • aside : indique une sidebar
  • figure
  • figcaption
  • time : indique une date de deux façons; pour l'ordinateur et pour l'homme
  • mark : met en valeur du texte

Exemple

<h2>Les articles</h2>
<nav>
    <ul>
        <li>Accueil</li>
        <li>Détails</li>
        <li>Contacts</li>
    </ul>
</nav>
<article>
    <header>
        <hgroup>
            <h1>Capgemini et HTML5</h1>
            <h2>Une percée prometteuse ?</h2>
        </hgroup>
    </header>
    <time datetime="2011-07-07">aujourd'hui</time>
    <p>Blah blah blah</p>
</article>

Nouveaux inputs

Arrivent également de nouveaux attributs / options

  • autocomplete : pour autoriser ou non l'auto-completion par le navigateur
  • autofocus : pour placer le curseur dans ce champ au chargement
  • list : pour lier à une datalist pour l'auto-completion
  • placeholder : pour afficher un texte qui se retire lors du focus
  • pattern : pour assurer que le contenu respecte un pattern
  • required : pour que le champ soit obligatoire

Tout ceci permet au navigateur (particulièrement mobiles) d'offrir un comportement plus adapté au type de champ. Cela permet aussi une uniformisation du comportement en le déléguant au navigateur plutôt qu'au développeur.

Nouvelles relations

On retrouve de nouvelles valeurs pour l'attribut rel (relation), des balises a et link. En voici une liste non-exhaustive.

  • archives : collection d'archives
  • author : indique plus d'informations sur l'auteur

Microdata

Les microdatas permettent d'annoter le DOM (Document Object Model : les objets décrits par les balises), avec des paires nom/valeur issues d'un vocabulaire personnalisé.

Problème : on peut représenter du texte, tableaux..., mais pas des personnes, factures ou autre.

Solution : utiliser les micro-datas et du vocabulaire personnalisé pour permettre cette représentation

Pourquoi : pour améliorer l'interaction avec les navigateurs (pouvoir enregistrer un contact en vCard, un évènement...), ou avec les moteurs de recherche.

Mode d'emploi

  1. Trouver le vocabulaire qui nous intéresse sur www.data-vocabulary.org si il a déjà été défini (Person, Event...) ou définir son propre vocabulaire.
  2. Déclarer le type et le scope (portée) du vocabulaire. Exemple : 
    <section itemscope itemtype="http://data-vocabulary.org/Person">
  3. Pour chaque valeur, indiquer son nom : 
    <h1 itemprop="name">Quentin Decré</h1>

Un exemple complet

<section itemscope itemtype="http://www.data-vocabulary.org/Person/">
	<h1 itemprop="name">Quentin Decré</h1>
	<h2 itemprop="title">Etudiant ingénieur INSA</h2>
	<a itemprop="url" href="www.quentin-decre.fr">Mon site</a>
	<p itemprop="address"
		itemscope itemtype="http://data-vocabulary.org/Address">
		<span itemprop="locality">Cesson-Sévigné</span>,
		<span itemprop="postal-code">35510</span>
		<span itemprop="country-name">France</span>
	</p>
</section>

Note : La propriété "adress" d'une personne est en une propriété complexe décrite par une autre micro-data.

Micro-data : Evaluation

  • Fiabilité : 100%
  • Important : Il n'y a aucune raison de ne pas utiliser les micro-data. Les navigateurs ne les gérant pas vont simplement les ignorer.

Application Cache

HTML5 permet de mettre en cache un ensemble de fichiers indiqués dans un manifest, et de les utiliser pour diminuer les chargements et accéder au site de façon déconnecté.

Quoi :

  • Un manifest
  • Une référence au manifest dans le code HTML des pages
  • Une gestion facultative des évènements de mise à jour

Pourquoi :

  • Réduire le chargement des pages
  • Précharger des pages
  • Accès déconnecté aux pages

Le manifest

3 parties :

  • Fichiers à mettre en cache
  • Fichiers pour lesquels il faut interdire la mise en cache (facultatif)
  • Page de substitution pour les pages non mises en cache (facultatif)

FALLBACK:
URLPattern Page

URLPattern ou Fichier
...

Fichier

Fonctionnement de la mise en cache

3 parties :

  • Le navigateur regarde si un manifest est indiqué dans la page, et si oui :
    • Il met en cache tout ce qui est indiqué dans la partie CACHE
    • Il retient ce qu'il ne faut pas mettre en cache (dans la partie NETWORK)
  • Il met en cache la page visitée (si elle n'est pas indiquée comme à ne pas mettre en cache)

Le manifest : exemple 1

Dans l'exemple suivant, pour toutes les pages (url pattern "/"), on affiche la page offline.html si elle n'est pas dans le cache. On met en cache toutes les pages indiquées dans cache, et on interdit la mise en cache des pages situées dans NETWORK.

CACHE MANIFEST
FALLBACK:
/ /offline.html
NETWORK:
/online/
CACHE:
/offline/page1.html
/offline/illustration.jpg
/offline/fichier.pdf

Le manifest : exemple 2

Dans cet exemple, le navigateur va mettre en cache uniquement les pages visitées. Le caractère * dans NETWORK indique que tout ce qui n'est pas dans le cache du navigateur peut être téléchargé depuis l'adresse originelle. Les images, objets incorporés dans les pages visitées seront ainsi récupérés en même temps que la page.

CACHE MANIFEST
NETWORK:
*

Relier une page au manifest

Pour relier un manifest à une page, il suffit d'ajouter l'attribut manifest dans la balise HTML, et en lui donnant l'URL du cache en valeur.

<html manifest="cache.appcache">

Les évènements

Des évènements sont déclenchés :

  • checking : lors de la vérification (que la page ait déjà été visitée ou non)
  • downloading : lors qu'un téléchargement vers le cache est lancé
  • progress : déclenché régulièrement pour informer sur l'avancement
  • cached : indique que le téléchargement est terminé
  • updateready : indique qu'il est possible de remplacer l'ancien cache pour le nouveau. Pour faire cette mise à jour, il suffit d'appeler window.applicationCache.swapCache();
  • error : Si le cache n'existe pas (404 ou 410), echec de téléchargement d'un ressource...

Note : si on n'intercepte pas d'évènement, cela n'empêche pas le navigateur de faire les mises à jour du cache.

  • Problème : un long script bloque l'interaction avec la page

Cette bonne pratique permet d'utiliser d'ores et déjà cette fonction

La vidéo de démonstration est proposée dans les formats mp4, webm, ogv, et si la balise vidéo n'est pas supportée, c'est un lecteur flash standard qui sera utilisé. Enfin, si rien ne fonctionne, un lien pour le téléchargement est donné.

Retour sur les objectifs d'HTML5

  • Unification du comportement du navigateur : ?... Internet Explorer commence à rattraper et semble prêt à se conformer aux exigences de cette nouvelle spécification. HTML5 n'étant pas finalisé, on trouve encore des différences. C'est donc un point à surveiller.
  • Normalisation : ?... Là encore, ce n'est pas terminé, et il est donc impossible de se prononcer, mais cela semble en bonne voie.
  • Nouvelles fonctions : Oui... La liste des nouvelles fonctions est conséquente, et nous allons voir celles qui apportent le plus.

Retour sur les fonctions d'HTML5

Les fonctions clés :

  • Offline Storage (cache, bases locales, stockage sur un modèle clé/valeur...) :une application web fonctionnant sans connection, et moins/pas dépendante du serveur.
  • WebSockets : connexion durable entre un serveur et une page web.
  • WebWorkers : possibilité de faire des taches lourdes sans paralyser une page.
  • Canvas : une surface 2D/3D entièrement programmable : environements 3D, modélisation, jeux, ...
  • OS Integration : accès à des informations clés (gyroscope, GPS...), mais aussi le glisser-déposer...

Remarques

  • Amélioration/accélération du développement : nouveaux éléments sémantiques, nouveaux attributs de style CSS... Cela évite le recours aux images, aux "astuces" pour certaines mises en formes,...
  • Alternative viable à flash (à terme ?) : Adobe lance edge (application de production d'animations HTML5), mais aussi flash player 11 molehill, supportant l'accélération graphique.
  • Alternative à un client lourd dans certains cas : encapsulation dans un client lourd (intel Encapsulator), ou grace au cache.
  • Compatibilité accrue : HTML5 sera un standard pour tous les types de navigateurs modernes existants quelque soit la plateforme.IE, bien qu'un peu en retard, est à l'écoute de la spécification et commence déjà à l'implanter dans IE10 (certaines fonctions sont même déjà dans IE9).

HTML5 : dès maintenant ?

La spécification d'HTML5 n'est pas encore fixée. L'utiliser dès maintenant ? ...

  • Oui, sans conditions pour certaines fonctions, dont le fallback est sans conséquences graves (nouveaux inputs types par exemple).
  • Oui avec conditions pour les autres fonctions :
    • Si il y a maitrise du navigateur des visiteurs
    • Après analyse de la maturité des fonctions utilisées
    • En s'appuyant sur des API au maximum pour lisser les différences et changements possibles.
